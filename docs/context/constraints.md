# Ограничения и регуляторные требования

> Все артефакты, генерируемые агентами, **обязаны** соблюдать каждое ограничение из этого документа. Нарушение любого ограничения = блокировка артефакта + эскалация на человека.

---

## C-001: Обязательная SCA для электронных платежей
- **Источник:** PSD2 (Directive 2015/2366), RTS on SCA
- **Правило:** Все электронные платежи в EEA требуют Strong Customer Authentication (минимум 2 из 3 факторов: знание, владение, биометрия), кроме применимых исключений (MIT, TRA, low value <€30, trusted beneficiary)
- **Для агента:** Любой артефакт, описывающий платёжный flow, ОБЯЗАН включать шаг SCA/3DS. При описании исключений — указать конкретное основание (Article 10-18 RTS)
- **Нарушение:** Блокировка артефакта → эскалация на Compliance Officer

## C-002: Запрет хранения чувствительных данных карт
- **Источник:** PCI DSS v4.0, Requirement 3
- **Правило:** Запрещено хранить полный PAN, CVV/CVC, PIN, магнитную полосу после авторизации. PAN в хранении — только токенизированный или маскированный (первые 6, последние 4)
- **Для агента:** В API спеках, data dictionary, логах — PAN только маскированный (411111******1111) или токен. CVV НИКОГДА не появляется в response, логах, БД. В sequence диаграммах указывать момент токенизации
- **Нарушение:** Немедленная блокировка → эскалация на CISO + PCI QSA

## C-003: Идентификация UBO при онбординге
- **Источник:** AMLD5 (Directive 2018/843), Article 30
- **Правило:** При онбординге юр. лица обязательна идентификация всех UBO с долей ≥25% (прямой или косвенной). Каждый UBO проходит KYC + sanctions screening + PEP check
- **Для агента:** Артефакты онбординга ОБЯЗАНЫ включать шаг UBO identification. User story онбординга без UBO check — невалидна. Порог 25% — жёсткий, не изменяемый
- **Нарушение:** Блокировка артефакта → эскалация на MLRO

## C-004: Sanctions screening при онбординге и ongoing
- **Источник:** EU Regulation 2580/2001, OFAC, UN Security Council Resolutions
- **Правило:** Обязательный скрининг по санкционным спискам (EU Consolidated, OFAC SDN, UN, UK HMT) при онбординге и не реже 1 раза в сутки для всей базы. Match = заморозка операций
- **Для агента:** Любой flow онбординга включает sanctions screening. Любой артефакт мониторинга включает daily batch screening. При match — не "уведомление", а немедленная блокировка + эскалация на MLRO
- **Нарушение:** Блокировка → эскалация на MLRO + Legal

## C-005: Сроки обработки чарджбэков
- **Источник:** Visa Core Rules (Section 11), Mastercard Chargeback Guide
- **Правило:** Visa: 30 календарных дней на representment с даты получения chargeback. Mastercard: 45 календарных дней. Pre-arbitration: 30 дней (Visa), 45 дней (Mastercard). Пропуск дедлайна = автоматический проигрыш
- **Для агента:** Артефакты dispute flow ОБЯЗАНЫ включать таймеры дедлайнов. Все сроки — в календарных днях от даты получения. Включать alert за 5 дней до дедлайна
- **Нарушение:** Блокировка → эскалация на Disputes Manager

## C-006: Порог отчётности по AML
- **Источник:** AMLD5, национальное законодательство (страна регистрации)
- **Правило:** Транзакции ≥€15,000 (или эквивалент) подлежат enhanced мониторингу. Серия связанных транзакций, суммарно превышающих порог за 24ч — также. Подозрительная активность — SAR в FIU в течение рабочего дня
- **Для агента:** В transaction monitoring артефактах: порог €15,000 жёстко зафиксирован. Включать агрегацию по merchant + 24ч окно. SAR flow — только manual (MLRO), агент НЕ автоматизирует подачу
- **Нарушение:** Блокировка → эскалация на MLRO

## C-007: GDPR — минимизация и retention
- **Источник:** GDPR (Regulation 2016/679), Articles 5, 17, 25
- **Правило:** Сбор только необходимых персональных данных. Retention period: транзакционные данные — 5 лет (AML), маркетинговые данные — до отзыва consent, логи — 1 год. Right to erasure применяется с учётом AML retention
- **Для агента:** Data dictionary обязан включать колонку retention period. API спеки — минимальный набор полей. Не проектировать хранение данных "на всякий случай". При конфликте GDPR vs AML — AML retention приоритетнее
- **Нарушение:** Блокировка → эскалация на DPO

## C-008: SLA по доступности процессинга
- **Источник:** Внутренний SLA, договоры с мерчантами
- **Правило:** Uptime авторизационного сервиса: ≥99.95% (не более 4.38 ч простоя/год). Latency p95 авторизации: ≤200ms. Latency p99: ≤500ms. Время онбординга мерчанта: ≤48ч (при прохождении KYB)
- **Для агента:** Любая архитектурная диаграмма включает failover/redundancy. API спеки включают timeout и retry policy. User story онбординга — SLA 48ч
- **Нарушение:** Блокировка → эскалация на CTO

## C-009: Логирование и audit trail
- **Источник:** PCI DSS v4.0 Requirement 10, AMLD5
- **Правило:** Все действия с платёжными данными логируются: кто, что, когда, откуда. Логи неизменяемы (append-only), хранятся ≥1 год онлайн + ≥5 лет в архиве. Доступ к логам — только авторизованный персонал
- **Для агента:** Каждый API endpoint включает audit log entry. Sequence диаграммы показывают запись в audit log. Логи НЕ содержат PAN, CVV, пароли — только маскированные/токенизированные значения
- **Нарушение:** Блокировка → эскалация на CISO

## C-010: Разделение среды (PCI DSS scoping)
- **Источник:** PCI DSS v4.0, Requirement 1, 2
- **Правило:** CDE (Cardholder Data Environment) изолирована от не-CDE. Все компоненты, обрабатывающие PAN, находятся в CDE. Сегментация подтверждена пенетрационным тестированием
- **Для агента:** Архитектурные диаграммы ОБЯЗАНЫ показывать границы CDE. Компоненты вне CDE НЕ обращаются к PAN напрямую — только через токенизацию. Любой новый сервис — определить in-scope/out-of-scope
- **Нарушение:** Блокировка → эскалация на CISO + PCI QSA

## C-011: Отчётность перед регулятором
- **Источник:** PSD2 Article 96, ECB requirements
- **Правило:** Инцидент безопасности, затрагивающий платёжные услуги — уведомление регулятора в течение 4 часов (initial), детальный отчёт в течение 3 рабочих дней. Fraud reporting — ежеквартально
- **Для агента:** Incident response flow включает timeline уведомлений. Не автоматизировать отправку регулятору — только подготовку отчёта
- **Нарушение:** Блокировка → эскалация на Compliance Officer + Legal

## C-012: Rate limiting и защита API
- **Источник:** PCI DSS v4.0 Requirement 6, OWASP API Security Top 10
- **Правило:** Все публичные API endpoints защищены rate limiting. BIN lookup: ≤10 req/min. Authorization: ≤100 req/sec per MID. Onboarding: ≤5 req/min per IP. Brute force protection на auth endpoints
- **Для агента:** Каждая API спека ОБЯЗАНА включать секцию Rate Limits с конкретными значениями. 429 Too Many Requests — обязательный error code
- **Нарушение:** Блокировка → эскалация на Tech Lead + Security
