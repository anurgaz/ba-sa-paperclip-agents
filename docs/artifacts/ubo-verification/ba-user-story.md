# User Story: Автоматическая верификация UBO через KYC провайдер

**ID:** US-001  
**Статус:** [DRAFT]  
**Тип:** Feature  
**Приоритет:** High  
**Epic:** Merchant Onboarding  
**Constraints:** C-003, C-004, C-007  

## User Story

**Как** Compliance Officer  
**Я хочу** автоматически верифицировать всех UBO с долей ≥25% через KYC провайдер при онбординге мерчанта  
**Чтобы** обеспечить соблюдение требований AMLD5 по идентификации конечных бенефициарных владельцев и ускорить процесс онбординга без компромисса в compliance

## Acceptance Criteria

### Happy Path
**AC-001:** При успешной KYC верификации всех UBO
- **Дано:** Мерчант прошёл KYB и предоставил данные всех UBO с долей ≥25%
- **Когда:** Система отправляет данные UBO в KYC провайдер
- **Тогда:** 
  - Система получает статус "VERIFIED" для каждого UBO
  - Статус онбординга обновляется на "UBO_VERIFICATION_COMPLETE"
  - Создаётся audit log запись с результатами верификации
  - Система переходит к следующему шагу онбординга

### Edge Cases
**AC-002:** При частичной верификации UBO
- **Дано:** 3 UBO, где 2 прошли верификацию, 1 требует дополнительные документы
- **Когда:** KYC провайдер возвращает статус "PENDING_DOCUMENTS" для одного UBO
- **Тогда:**
  - Система помечает онбординг как "UBO_VERIFICATION_PENDING"
  - Отправляется уведомление мерчанту с запросом недостающих документов для конкретного UBO
  - Создаётся таск для Compliance Officer для review pending case
  - Устанавливается таймер напоминания через 72 часа

**AC-003:** При изменении долей владения во время верификации
- **Дано:** UBO A имел 30%, во время верификации доля снизилась до 20%
- **Когда:** Система обнаруживает изменение структуры собственности
- **Тогда:**
  - Система останавливает верификацию данного UBO
  - Пересчитывает список актуальных UBO ≥25%
  - Запускает верификацию только для актуального списка
  - Создаёт alert для Compliance Officer об изменении структуры

### Error Scenarios
**AC-004:** При отказе KYC провайдера в верификации
- **Дано:** KYC провайдер возвращает статус "REJECTED" для одного из UBO
- **Когда:** Система получает отказ с reason code
- **Тогда:**
  - Онбординг блокируется со статусом "UBO_VERIFICATION_FAILED"
  - Создаётся high-priority alert для MLRO
  - Мерчанту отправляется уведомление о невозможности завершить онбординг
  - Инициируется санкционный скрининг для проверки причины отказа (C-004)
  - Система не позволяет активировать мерчанта

**AC-005:** При технической недоступности KYC провайдера
- **Дано:** KYC провайдер недоступен (timeout/500 error) более 30 секунд
- **Когда:** Система не может получить ответ после 3 retry попыток
- **Тогда:**
  - Онбординг переводится в статус "UBO_VERIFICATION_TECHNICAL_ERROR"
  - Создаётся incident ticket для Tech Support
  - Compliance Officer получает уведомление о техническом сбое
  - Система автоматически повторяет запрос каждые 15 минут до 24 часов
  - Если не восстановлено за 24 часа - эскалация на manual review

## Business Rules
- **BR-001:** Порог UBO 25% — жёстко зафиксирован согласно AMLD5 (ссылка на C-003)
- **BR-002:** Все UBO обязательно проходят sanctions screening перед KYC (C-004)
- **BR-003:** Timeout KYC запроса: 30 секунд, retry policy: 3 attempts with exponential backoff
- **BR-004:** Персональные данные UBO хранятся согласно GDPR retention policy (C-007)

## Definition of Done
- [ ] API интеграция с KYC провайдером реализована
- [ ] Обработка всех статусов ответов KYC провайдера (VERIFIED, PENDING, REJECTED, ERROR)
- [ ] Audit logging всех KYC запросов и ответов
- [ ] Unit тесты покрывают все AC
- [ ] Integration тесты с mock KYC провайдером
- [ ] Документация API обновлена
- [ ] Compliance review пройден
- [ ] Security review пройден (данные UBO в transit/at rest)

## Technical Notes
- Интеграция через REST API KYC провайдера
- Асинхронная обработка для множественных UBO
- Encryption персональных данных при передаче (TLS 1.3 minimum)
- Токенизация/маскирование PII в логах

## Dependencies
- Актуальный список UBO получен и валидирован на предыдущем шаге онбординга
- KYC провайдер выбран и контракт подписан
- Sanctions screening module интегрирован

2. Нужна ли возможность manual override для edge cases?
3. Требуется ли уведомление UBO о прохождении верификации (privacy implications)?
